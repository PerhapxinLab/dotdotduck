# v0.2.0 · Task Agent + WebAgent multi-instance + shared sessions

Design note for the Wave 2 portion of the v0.2.0 development cycle.
Updates the original Track 2 plan (`mode-based webagent`) — the
mode-based design was reworked after the user pointed out it
violated the existing "one class per agent kind" pattern that
`InlineAgent` already followed.

Status: planned. No code yet — when each sub-wave lands it gets
flipped to ✅ here and in `README.md` v0.2 entry.

## What changed and why

The earlier plan put `chat` mode and `operate` mode inside the same
`WebAgent` class. That conflicted with how the rest of the SDK is
shaped:

- `WebAgent` does one thing: DOM-grounded page operation.
- `InlineAgent` does one thing: edit selected text in editables.

Cramming chat + operate into one class would have:

- Sprayed `if (mode === 'chat') skip` branches through the runtime.
- Forced one class to understand two protocols (CoT envelope vs
  plain tool-calling chat).
- Made the live registry API ambiguous — which mode does this tool
  belong to?
- Made adding a third agent kind later painful.

The correct refactor: a NEW agent class — `TaskAgent` — that lives
beside `WebAgent` and `InlineAgent`. Same SDK pattern, three kinds
of agent, no internal mode switching.

## What we keep, what we drop

From the original Track 2 plan:

| Item | Decision | Reason |
| --- | --- | --- |
| `protocol: 'plain' \| 'cot'` enum on WebAgent | **Drop** | Plain protocol lives on TaskAgent. WebAgent stays CoT-only. |
| `enter_mode` tool | **Drop** | Cross-agent delegation, if needed, is a host-router concern (palette routes the query to TaskAgent or WebAgent). |
| `agent_mode_changed` event | **Drop** | No internal modes to switch. |
| Same-turn intercept logic | **Drop** | Goes away with `enter_mode`. |
| Live registry (`registerTool` / `registerContextProvider`) | **Keep** | Useful for WebAgent on its own — host can add tools / providers mid-session. |
| Context providers split | **Keep** | Lets host replace DOM dump logic, screenshot logic, history serialization. |
| `agent_tool_failed` event | **Keep** | Orthogonal to mode/agent — fills the visibility gap where `agent_tool_end` only fires on success today. |
| `onLoopEnd` hook | Already shipped | ✅ in this v0.2 cycle, Wave 1·B (commit `c556c9d`). |

Net new:

- **`TaskAgent` class** — plain protocol + tool calling, no DOM.
- **`WebAgent` multi-instance** — host can register named instances
  (one per route / persona / role).
- **`AgentSession` is shareable** — multiple agents (even of
  different kinds — see "cross-type" caveat below) can be wired to
  the same `Session`, so cross-page conversation history + memory
  carry over.

## Architecture: three agent kinds

```text
WebAgent     DOM-grounded page operation
             CoT envelope (memory / next_goal / actions[])
             actions: navigate, click, fill_input, narrate, ...
             subtitle bar + cursor, Space-gating

InlineAgent  in-place editing of selected text
             single-shot LLM call (or tool-loop) per action
             menu: Translate / Improve / Rewrite / ...
             no session, no narration

TaskAgent    conversation + tool calling
             plain protocol (OpenAI tool-call chat)
             actions: host-defined tools only (lookup CRM, RAG, ...)
             optional narration to subtitle; no Space-gating
             can SHARE session with WebAgent (same `Session`)
```

The three are composable — a host wires whichever surfaces they
need. A SaaS product might wire all three: WebAgent for page
operation, InlineAgent for the doc editor, TaskAgent for the
customer-service Q&A box.

## Sub-waves

### 2·A · Shareable sessions + agent registry

**Goal**: let a host inject the same `AgentSession` into multiple
agents, and route the "active" agent per page without losing
history.

**API shape**:

```ts
// Today (singleton-ish — one WebAgent at orchestrator level):
const dddk = new DotDotDuck({ webAgent: { llm, sitemap, systemPrompt, ... } });

// After 2·A:
const session = dddk.sessions.get('main');     // create-or-fetch named session

const docsAgent = new WebAgent({
  llm: nano,
  sitemap: docsSitemap,
  systemPrompt: 'You help docs visitors find pages.',
  session,                                      // <- inject
});

const adminAgent = new WebAgent({
  llm: mini,
  sitemap: adminSitemap,
  systemPrompt: 'You help admins manage tenants.',
  session,                                      // <- same session
});

dddk.agents.register('docs',  docsAgent);
dddk.agents.register('admin', adminAgent);
dddk.agents.setActive('docs');                  // current page is /docs

// Later, user navigates to /admin:
dddk.agents.setActive('admin');
// adminAgent picks up `session.turns[]`, sees the docs conversation
// + memory, continues with its own persona / sitemap.
```

**Files**:

- `src/agent/webagent/session.ts` — already exists, no shape change
  needed.
- `src/agent/webagent/webagent.ts` — extend `WebAgentConfig` to
  accept `session?: AgentSession`. If absent, agent creates its own
  (current behavior).
- `src/orchestrator/sessions-registry.ts` (new) — `dddk.sessions`
  namespace, named session cache, persistence wiring.
- `src/orchestrator/agents-registry.ts` (new) — `dddk.agents`
  namespace, named instance registry, `setActive(name)`,
  `getActive()`.
- `src/orchestrator/index.ts` — wire both namespaces onto the
  `DotDotDuck` handle.

**Compat**: the existing single-WebAgent path stays default. If host
doesn't touch `dddk.sessions` / `dddk.agents`, the orchestrator
behavior is identical to v0.1.

**Cross-type session note**: a single `Session.turns[]` can hold
WebAgent (CoT-shape) and TaskAgent (plain-shape) turns. The schema
allows it; the re-serialization cost when one agent reads the
other's turns is non-trivial. **Scope for 2·A: same-type sharing
only** — WebAgent ↔ WebAgent or TaskAgent ↔ TaskAgent. Cross-type
sharing is a v0.3 item.

### 2·B · TaskAgent class

**Goal**: a new agent class for conversation + host-defined tool
calling. Independent of WebAgent.

**API shape**:

```ts
import { TaskAgent } from '@perhapxin/dddk/agent';

const taskAgent = new TaskAgent({
  llm: nano,
  systemPrompt: 'You answer customer-service questions for Acme Co.',
  tools: [
    {
      name: 'lookup_order',
      description: 'Fetch order status by order id.',
      parameters: { type: 'object', properties: { id: { type: 'string' } } },
      handler: async ({ id }) => fetch(`/api/orders/${id}`).then((r) => r.json()),
    },
    // ...
  ],
  // Optional — share with WebAgent for cross-page continuity.
  session,
  // Optional — narrate progress to the subtitle bar like InlineAgent does.
  enableSubtitle: false,
  // Optional — cap on a single turn's tool-call loop.
  maxToolRounds: 5,
});

taskAgent.attachTo(dddk);

// Usage:
await taskAgent.ask('Where is my order #12345?');
// LLM calls lookup_order → gets data → composes answer → narrates / returns.
```

**Properties**:

- **No DOM read**. Doesn't dump pages, doesn't read selectors.
  Context comes purely from the conversation + tool returns.
- **No CoT envelope**. Standard `messages: [{role, content}]` +
  standard OpenAI tool calls. The LLM responds with either text
  (final answer) or a tool call (loop continues).
- **No subtitle gating by default**. TaskAgent's UX is "ask and
  receive" — no Space-gating between tool calls. Hosts who want
  step-by-step narration enable `enableSubtitle`.
- **Session-compatible**. Can share `AgentSession` with another
  TaskAgent for cross-page continuity. (Cross-type with WebAgent is
  v0.3.)
- **Streaming**. Same `streamComplete` path as WebAgent — final
  text streams to the host as it arrives.

**Files**:

- `src/agent/task/index.ts` — `TaskAgent` class, public API.
- `src/agent/task/runtime/loop.ts` — tool-call loop (re-uses
  existing `executePlainLoop` shape if anything's reusable; more
  likely a focused 200-line implementation).
- `src/agent/task/types.ts` — `TaskAgentConfig`, `TaskTool`,
  `TaskRunOptions`.
- `src/agent/task/task.test.ts` — smoke tests (single ask,
  tool-loop, session share, max rounds).
- Re-export from `src/agent/index.ts` and `src/index.ts`.

**Out of scope for 2·B**:

- Cross-type session re-serialization (v0.3).
- Built-in tool catalog (TaskAgent ships with zero tools — every
  tool is host-defined).
- Multi-agent delegation (`TaskAgent` calling `WebAgent` — also
  v0.3 if there's appetite).

### 2·C · Live registry on WebAgent

**Goal**: host can `registerTool` / `registerContextProvider` after
WebAgent construction. In-flight turn snapshots; next turn sees the
new entry.

**API shape**:

```ts
const docsAgent = new WebAgent({ llm, sitemap, systemPrompt, tools: [...] });

// Later — host wires a new feature:
const handle = docsAgent.registerTool({
  name: 'open_billing',
  description: 'Open the billing page and highlight the upgrade CTA.',
  parameters: { ... },
  handler: ({ tier }) => navigateAndHighlight('/billing', `.tier-${tier}`),
});

// Remove if the feature is unmounted:
handle.remove();
```

**Files**:

- `src/agent/webagent/tools-registry.ts` — already exists (per
  earlier roadmap item 1.6 `ToolsRegistry.flush()`). Add
  `registerTool` / `registerContextProvider` surface here.
- `src/agent/webagent/webagent.ts` — expose
  `registerTool(def) → handle`, `registerContextProvider(name, fn) → handle`.

### 2·D · Context providers split

**Goal**: replace the monolithic DOM-dump + screenshot + history +
selection logic with named providers the host can replace.

**API shape**:

```ts
// SDK ships defaults — host doesn't need to wire any.
// To override:
docsAgent.setContextProvider('dom', async (ctx) => {
  // Custom DOM serialization — e.g. only emit elements inside a specific region.
  return customDomDump(ctx);
});

docsAgent.setContextProvider('screenshot', async (ctx) => {
  // Custom screenshot — e.g. host renders an SVG diagram via canvas.
  return canvasToDataURL();
});
```

Default providers in the SDK:

- `url` — current path + query + hash
- `page_summary` — `<title>` + meta description
- `dom` — current compressed DOM dump (existing logic)
- `screenshot` — `getBoundingClientRect` of viewport, image data URL
- `history` — last N navigations
- `selection` — current `window.getSelection()`

**Files**:

- `src/agent/webagent/context-providers/` (new) — one file per
  default provider.
- `src/agent/webagent/webagent.ts` — add
  `setContextProvider(name, fn)` and `registerContextProvider(name, fn)`
  (alias kept for symmetry with `registerTool`).

### 2·E · `agent_tool_failed` event

**Goal**: emit a typed event when a tool handler throws or returns
`{ ok: false }`. Currently the only signal is `agent_tool_end` on
success, leaving failures invisible to analytics.

**API shape** (additive to `IntentEvent` union):

```ts
| {
    kind: 'agent_tool_failed';
    runId: string;
    name: string;          // tool id
    reason: string;        // err.message or { ok: false, reason }
    attempt: number;       // retry count when applicable
    timestamp: number;
  }
```

**Files**:

- `src/types.ts` — add to `IntentEvent` union.
- `src/agent/webagent/runtime/loops.ts` (or wherever tool dispatch
  is) — emit on the failure paths.
- `dddk-frontend/src/lib/analytics-sanitize.ts` — add the new event
  shape so it survives sanitisation.

## Sequencing

```text
2·A  Sessions + agent registry          1–1.5 d   (foundation)
2·C  Live registry on WebAgent          0.5 d
2·D  Context providers split            1 d
2·E  agent_tool_failed event            0.5 d
2·B  TaskAgent                           1.5–2 d   (leans on A + D)
```

Total ~4.5–5.5 days. Each sub-wave commits + pushes independently
so the working tree is shippable at every step.

## Out-of-scope reminders

These were considered and explicitly deferred:

- **Cross-agent-type session sharing** — TaskAgent and WebAgent
  reading each other's turns. Real value but the re-serialization
  cost (CoT ↔ plain) is non-trivial. v0.3.
- **Multi-agent orchestration / delegation** — `TaskAgent` calling
  `WebAgent` via a tool. Workable but introduces orchestrator
  complexity. v0.3 if needed by a real use case.
- **Cross-tenant memory / RL trajectory aggregation** — that's
  squarely a dddk-console (paid) responsibility, not the OSS SDK.
- **Built-in TaskAgent tool catalog** — every host's tool needs are
  different. Ships with zero, hosts bring their own.
